## 한 문장 진단

**이 7개는 "완성된 영상 예시"가 아니라 명소·주제별 "영상 제작 스토리보드(샷 리스트 + 촬영 설정 + 동선 + 편집 흐름 + BGM)"다. 즉 완성작 쇼케이스가 아니라 촬영 계획·편집 설계 도구로 써야 정직하고, 그렇게 쓰면 4부 현장 워크플로와 1·2·3부 명소 페이지를 잇는 매우 강한 연결 조직이 된다.**

## 2. 반영 전에 반드시 짚을 4가지 (정직성·정합성)

1. **콘셉트 이미지 문제 (가장 중요).** 각 컷 썸네일은 제 판독으로는 **실사 촬영본이 아니라 기획용 콘셉트/일러스트 이미지로 보입니다(미확인)**. 어느 쪽이든 저자의 실제 몽골 촬영본이 아님은 분명합니다(트립 8/13, 아직 미래). → 책의 기존 정직성 라벨 규약(2부 `capcut-storyboard.md`의 "대표 시나리오 · 실촬영본은 트립 후 교체")을 **그대로 승계**해야 합니다. "완성 영상 예시"로 소개하면 안 됩니다.
2. **장비 불일치.** 스토리보드는 **DJI Mini 4 Pro**로 적혀 있으나 이 책의 드론은 **Mini 5 Pro**입니다. RS 3 Mini 짐벌·iPhone 15 Pro도 등장합니다(책 지상 카메라는 R7로 일치 ✓). → 설정값(fps·ND·비행고도)은 Mini 5 Pro 기준 **재확인 표기** 필요, 또는 "Mini 4 Pro 기준 초안 — 5 Pro는 수치 재확인" 명시.
3. **명소 불일치.** 스토리보드에 **어르헝(오르혼 밸리, UNESCO·중부 몽골)**이 있는데 이 책 명소 5곳(차강소브라·바양작·홍고린엘스·욜링암·바가가즈링촐로)에 **없습니다**. 반대로 **바양작(Bayanzag)은 스토리보드가 없습니다**. → 어르헝을 책 일정에 넣을지(코스 확장) 아니면 "보너스 명소"로 별도 취급할지 결정 필요. 바양작은 향후 보완 대상.
4. **이미지 용량.** PNG 각 ~2.2MB × 7 ≈ **16MB**. GitHub Pages 배포·로딩 부담. → 임베드하려면 기존 규약대로 `sips`로 ~2000px·JPEG 리사이즈(장당 수백 KB) 후 삽입.

## 3. 무엇에 쓸 것인가 — 용도 3층

- **주 용도 = 촬영 계획서(샷 리스트).** "이 명소에 가면 이런 컷을 이 순서·이 설정으로 담아 한 편을 만든다"는 **현장 촬영 가이드**. 4부 `preparation.md`(촬영 계획)·명소 페이지와 직결.
- **보조 용도 = 편집 설계 예시.** 편집 흐름·BGM·전환이 함께 있어 2부 CapCut / 3부 스태킹·타임랩스 편집의 **구체 청사진**.
- **캡스톤 용도 = 세 카메라 오케스트레이션의 실제 사례.** 통합형 3개(홍고린·욜링암·바가)는 4부 `field-day.md`의 "세 카메라 오케스트레이션"을 **한 명소에서 실현한 워크드 예제**. 4부의 추상 설명에 살을 붙인다.

## 4. 배치안 — 3가지 (A 권장)

### 안 A (권장) — 2부에 "명소별 영상 스토리보드" 챕터 신설 + 3부·4부 교차링크

- 드론 영상이 7개 중 5개의 공통분모이고, 기존 유일 스토리보드(`capcut-storyboard.md`)도 이미 2부에 있음 → **2부 드론 영상 하위에 신규 그룹** `2-drone/storyboards/`.
  - 드론 전용 3개(차강소브라·말떼·어르헝) = 드론 영상 스토리보드.
  - 통합 3개(홍고린·욜링암·바가) = "드론+지상 통합" 라벨 + **4부 field-day.md로 상향 링크**("세 카메라 오케스트레이션 실제 적용").
- 은하수 1개(밤하늘) = **3부 `6-bonus/timelapse.md`에 배치/교차링크**(은하수 타임랩스 영상 스토리보드).
- 각 명소 스토리보드는 해당 **명소 페이지(1·2·3부)에서 역링크** → 발견성 확보.
- 장점: 성격상 "드론 영상 제작물"이라는 정체성과 일치, 기존 스토리보드와 한곳에, 편집 파트와 인접. 단점: 통합·은하수는 교차링크로 보완.

### 안 B — 각 명소 페이지에 분산 삽입

- 차강소브라 스토리보드 → `2-drone/2-sites/tsagaan-suvarga.md` 등, 명소별로 흩어 삽입.
- 장점: 명소 문맥에서 바로 발견. 단점: 통합형·말떼·밤하늘·어르헝은 1:1 명소 매핑이 안 됨(어르헝은 페이지 자체가 없음), 샷 리스트라는 "한 편 만들기" 성격이 파편화.

### 안 C — 4부에 "영상 스토리보드" 통합 허브

- 4부 현장 워크플로가 세 트랙을 잇는 캡스톤이므로 스토리보드 7개를 4부에 모아 허브화.
- 장점: 가로 연결의 정점에 배치. 단점: 4부의 "과정(준비→하루→정리)" 정체성에 대용량 갤러리가 섞여 톤이 흐려짐.

## 5. 권장 실행 순서 (안 A 채택 가정)

1. **이미지 정직성·정합성 처리:** 7개를 `sips`로 ~2000px JPEG 리사이즈 → `src/images/storyboards/`. 각 캡션에 "기획용 콘셉트 스토리보드 · 실촬영/완성 영상은 트립(8/13) 후 교체 · 장비 표기는 Mini 4 Pro 초안이며 Mini 5 Pro 수치 재확인" 라벨.
2. **2부 신규 그룹 스텁 + SUMMARY 배선**(create-missing=false 유지, 빌드 그린).
3. **각 스토리보드 페이지 작성**: 스토리보드 이미지 + 그 명소/주제의 "촬영 계획" 텍스트(재설명 금지, 촬영·편집법은 기존 파트 링크 승계) + 교차링크.
4. **역링크 삽입**: 명소 페이지·4부 field-day.md·3부 timelapse에서 각 스토리보드로.
5. **어르헝·바양작 결정 반영**(코스 확장 vs 보너스 vs 보류).
6. **빌드·링크 게이트**(dead 0).

> 이 작업은 분량상 GSD **새 마일스톤(v1.6 "영상 스토리보드")** 또는 소규모 단발 페이즈로 진행 가능. 규모·정직성 처리(리사이즈·라벨) 때문에 마일스톤화를 권장.
